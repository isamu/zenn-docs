RSC（React Server Components）や GraphQL は「高度なスケール前提の課題」を解くための技術であり、多くの普通の会社・プロダクトにとっては過剰設計になりがちです。


## 1. そもそも RSC / GraphQL が解こうとしている「前提問題」

これらの技術は、**最初から以下のような前提**を置いています。

### RSC の前提

* 巨大なコンポーネントツリー
* クライアント JS の削減が事業上クリティカル
* サーバーとクライアントの境界を極限まで最適化したい
* FB/Meta クラスのトラフィック・人員・CI/CD

👉 **「UI = 分散システム」** という世界観

### GraphQL の前提

* クライアントが多数（Web / iOS / Android / 他社）
* API 進化のスピードが極端に速い
* REST のバージョニングが耐えられない
* 社内に API ガバナンス組織がある

👉 **「API = 社内プロダクト」** という世界観


## 2. 普通の会社で起きていること（現実）

多くの会社では、実際には：

* フロントエンド 1〜2種類
* バックエンドも数人
* データ構造は年に数回しか変わらない
* トラフィックも常識的な範囲
* デバッグできる人が限られている

この条件下で RSC / GraphQL を入れると：

### RSC の副作用

* 実行場所（Server / Client）が直感的でなくなる
* hydration / streaming / cache の理解が必須
* セキュリティ境界が曖昧になりやすい
* **「なぜ動いているか誰も説明できない」状態**

### GraphQL の副作用

* N+1 / resolver 地獄
* スキーマと実装の二重管理
* 型はあるが、**実行時エラーは増える**
* 単純な CRUD が逆に読めなくなる

👉 **「問題を解決するために、別の問題を大量に持ち込んでいる」**

## 3. なぜ「不要なのに OSS に入ってくる」のか

これは技術的というより **社会構造の問題**です。

### ① Big Tech の成功体験の輸出

* Facebook / Netflix / GitHub の課題は特殊
* それが **「モダンアーキテクチャ」** として一般化される
* コンテキストが削ぎ落とされる

### ② OSS は「複雑さを売る」インセンティブを持つ

* 単純な技術は差別化しづらい
* 複雑なほど「すごそう」に見える
* カンファレンス・登壇・採用に有利

### ③ フレームワークの競争

* 「新しい抽象」を出さないと存在感が出ない
* 結果として **抽象の上に抽象**が積み重なる


## 4. 本質的な問い：「それは誰のための複雑さか？」

ここが重要です。

| 観点      | 誰が得するか                |
| ------- | --------------------- |
| RSC     | フレームワーク作者 / 超大規模プロダクト |
| GraphQL | API プラットフォームを持つ企業     |
| 普通の会社   | **ほぼ得しない**            |

逆に普通の会社が本当に欲しいのは：

* 読めるコード
* 予測可能な実行モデル
* ログを見て原因がわかること
* 新人が 1 週間で理解できること


## 5. 「じゃあ何を使えばいいのか？」

### 多くのケースで十分な構成

* **REST + JSON**
* **Server / Client の明確分離**
* **ビルド時 or 明示的なデータ取得**
* **キャッシュは CDN / HTTP キャッシュで解決**

これで 9 割のプロダクトは壊れません。

## 6. 総括

> RSC や GraphQL は
> **「スケールした結果、仕方なく生まれた技術」** であって
> **「スケールしていない段階で入れるべき技術」ではない**

普通の会社がそれを採用するのは、

* まだ起きていない問題を恐れ
* 現在のシンプルさを犠牲にし
* 将来の保守性を下げる

という本末転倒になりがちです。
