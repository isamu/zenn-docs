「DRY にしよう」は、レビューで人間が守るのが一番向いていないルールです。目の前の差分は綺麗でも、**300 ファイル向こうに同じ関数がすでにある**ことは、レビュアーの記憶にしか頼れないからです。

実際、私たちの [MulmoClaude](https://github.com/receptron/mulmoclaude) では `truncate()` が**6 実装**に増えていたことがあります。誰も悪くない。ただ、人間の記憶が 26 万行をカバーできないだけです。

なら機械にやらせよう、ということで **jscpd** を実際のモノレポに当てて実測しました。結果、**CI ゲートの設計に関する落とし穴**が見えたので、その話まで書きます。

## ツールの棲み分け

まず「重複検出」と一口に言っても、レイヤが違うものが混ざりがちなので整理します。

| ツール | 何を見るか | 守備範囲 |
|---|---|---|
| **jscpd** | **トークン列レベルのコピペ**（ファイル横断） | 「同じコードが別の場所にある」を機械的に検出。本記事の主役 |
| **eslint-plugin-sonarjs** | 同一関数・同一条件式・認知的複雑度などのコードスメル | AST ベース。`no-identical-functions` など。ESLint に統合できる |
| **ts-prune / knip** | 使われていない export・デッドコード | DRY ではないが「余分なコード」を減らす |

重要なのは、**sonarjs と jscpd は競合ではなく補完**だという点です。sonarjs の `no-identical-functions` は「完全に同一の関数」を AST で見ますが、jscpd は**トークン列の一致**を見るので、少し変数名が違うコピペや、関数の一部だけのコピペ、ファイルを跨いだブロックまで拾えます。

## jscpd v5 は Rust になっていた

`jscpd` は v5 系で **Rust 実装**に移行しています（v4 系が TypeScript 実装）。

- 223 言語フォーマット対応
- 13 レポーター
- プリビルドバイナリ提供（Node ランタイム不要）
- `--threshold` 超過で exit 1 → **CI にそのまま組める**

```bash
npx jscpd@5 src server              # 実行するだけ
npx jscpd@5 src --threshold 5       # 5% 超えたら exit 1（CI が落ちる）
npx jscpd@5 . --blame -r console-full  # git blame で「誰のコピペか」まで
```

主なオプション:

| オプション | 既定 | 意味 |
|---|---|---|
| `--min-tokens` (`-k`) | 50 | クローンとみなす最小トークン数 |
| `--min-lines` (`-l`) | 5 | クローンとみなす最小行数 |
| `--mode` (`-m`) | `mild` | `mild` / `weak` / `strict` |
| `--format` (`-f`) | 全部 | 対象フォーマットを絞る |
| `--ignore-pattern` (`-i`) | — | 除外グロブ |
| `--threshold` (`-t`) | — | 超えたら exit 1 |

## 実測 — 26.7 万行のモノレポに当てる

対象は [MulmoClaude](https://github.com/receptron/mulmoclaude) — Claude Code / Codex をコアに据えた実プロダクトのモノレポ（Vue + TypeScript、`packages/` に 25 超のワークスペース）です。

```bash
npx jscpd@5 src server packages \
  -i "**/node_modules/**" -i "**/dist/**" -i "**/*.d.ts"
```

結果:

| 種別 | ファイル | 総行数 | クローン | 重複行 |
|---|---|---|---|---|
| **typescript** | 1,170 | 163,210 | 314 | 5,332 (**3.27%**) |
| markdown | 58 | 8,010 | 84 | 1,818 (22.70%) |
| json | 78 | 5,607 | 45 | 858 (15.30%) |
| javascript | 20 | 1,757 | 7 | 189 (10.76%) |
| css | 29 | 13,080 | 27 | 620 (4.74%) |
| html | 129 | 34,970 | 35 | 687 (1.96%) |
| **vue** | 132 | 35,088 | **0** | 0 (0.00%) |
| **Total** | **1,666** | **267,098** | **512** | **9,504 (3.56%)** |

**実行時間 127ms。** 26.7 万行を 0.1 秒台で舐めるので、CI に置くコストは実質ゼロです。ここは素直に感動しました。

### 気づき 1 — 「Total 3.56%」で語ってはいけない

言語別に分解すると景色が変わります。

- **markdown 22.70% / json 15.30%** — これは**ノイズ**です。json は 25 超のパッケージの `package.json` 定型（`scripts` や `devDependencies` が似るのは当たり前）、markdown は docs / skills。「重複」ではあるが、**共通化すべき対象ではない**。
- しかも markdown の重複トークンは **561.90%** と 100% を超えました。重複ブロックの多重カウント由来のアーティファクトと思われ、少なくともこの指標を markdown で信用してはいけない、という判断材料になります。
- **vue は 0 クローン**。SFC は構造上コピペしにくいのか、綺麗に 0 でした。

つまり **DRY ゲートとして見るべきは TypeScript の 3.27%** であって、Total の 3.56% ではない。`--format typescript` で絞るのが正解です。

### 気づき 2 — `packages/` を入れると数字が跳ねる

`src` + `server` だけだと TypeScript は **1.90%**（113 クローン）。`packages/` を足すと **3.27%**（314 クローン）に上がりました。

犯人は「似た定型の多いアダプタ群」です。25 個超の bridge パッケージ（Slack / Discord / Telegram / …）が、それぞれ似た初期化・送信・エラーハンドリングを持つのは**構造上ある程度自然**で、無理に共通化すると逆に読みにくくなる領域でもあります。

数字が上がったからといって即「悪」ではない。**どこの重複か**を見ないと意味がない、という当たり前のことが数字で出ます。

## 落とし穴 — グローバル % の閾値は「新規重複」を止められない

ここが本記事で一番言いたいところです。

`--threshold 5` を CI に置けば「DRY が守られる」と思いがちですが、**冒頭の `truncate()` が 6 実装に増える事故は、この閾値では止まりません。**

理由は算数です。16 万行に対して、新しく 20 行の `truncate()` をコピペしても、全体の重複率はほぼ動きません。3.27% が 3.28% になるだけ。**閾値は永遠に超えないまま、重複だけが静かに増えていきます。**

つまり:

- **グローバル % ゲート** = 「大崩れの保険」。安いので入れてよいが、**新規重複はほぼ素通り**する。
- 本当に効かせたいなら、**PR 差分で「新しく増えたクローン」を検出**する必要がある。母数に薄められない指標にしないと、ゲートとして機能しません。

閾値を入れて安心するのが一番危ない、というのが実測から得た結論でした。

## 使い方の三択

実測を踏まえて、こう整理しました。

| 方針 | 効果 | コスト |
|---|---|---|
| **(a) CI でグローバル % ゲート** | 大崩れの保険。127ms なので実質タダ | 低。ただし新規重複は止まらない |
| **(b) PR 差分で新規クローン検出** | `truncate()` 型の事故に効く**本命** | 実装が一手間 |
| **(c) スキル化してオンデマンド** | 「検出 → AI が共通化案」まで | 低。人が起動する必要あり |

(a) は入れるとしても「保険」と割り切る。効くのは (b)、手軽なのは (c)、という温度感です。

## AI 時代の使い方 — `ai` レポーターと MCP

ここが最近の jscpd の面白いところで、**LLM に食わせる前提の導線が公式にあります**。

```bash
npx jscpd@5 src --reporters ai
```

`ai` レポーターは「**トークン効率の良い重複レポート**」を出力します。人間向けの装飾を削って LLM に渡しやすくしたもので、これをそのまま

> 「この重複を共通関数に抽出して」

に繋げられます。さらに公式に **MCP サーバ**と、**`dry-refactoring` スキル**まで用意されています。

```bash
npx skills add https://github.com/kucherenko/jscpd --skill dry-refactoring
```

つまり **「検出は機械、抽出方針の提案は LLM、採否は人間」** の分業が、追加の自作なしで組める状態になっています。DRY 違反の検出は決定的（トークン一致）なので機械が強く、「これは共通化すべきか、それとも似ているだけか」は文脈判断なので LLM/人間が強い。役割分担として素直です。

先ほどの (b)（PR 差分で新規クローン検出）と (c)（AI に抽出案を出させる）を繋ぐと、**「PR で増えた重複だけを AI に渡してリファクタ案を出す」**という、母数に薄められない実用的なループが作れます。ここが次に試したいところです。

## まとめ

- **jscpd v5 は Rust 製で速い。** 26.7 万行 / 127ms。CI コストは実質ゼロ。
- **Total の重複率で語らない。** 言語別に分解すると markdown・json はノイズ。見るべきは TypeScript。
- **グローバル % 閾値は新規重複を止めない。** 母数に薄まるので、`truncate()` が 6 実装になる事故は素通りする。効かせるなら PR 差分。
- **sonarjs と jscpd は補完関係。** AST の同一関数検出とトークン列のコピペ検出は別物。
- **`ai` レポーター / MCP / `dry-refactoring` スキル**が公式にあり、「検出は機械・提案は LLM・採否は人間」の分業が既製品で組める。

「入れたら安心」ではなく「**何を止めたいのか**」を先に決めてから閾値を設計する。当たり前ですが、実測して初めて腹落ちしました。
